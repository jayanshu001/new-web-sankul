# WebSankul Postman Collections

Module-wise Postman v2.1 collections, generated from the route files in `src/admin` and `src/client`.

| File | Folders | Requests |
|------|--------:|---------:|
| `WebSankul-Admin.postman_collection.json` | 38 | 503 |
| `WebSankul-Client.postman_collection.json` | 43 | 243 |

## What's preset

1. **`{{BASEURL}}` variable** â€” no hardcoded `192.168.0.5`. Every request URL is `{{BASEURL}}/api/v1/admin/...` (or `/client/...`). Set `BASEURL` in a Postman **Environment** to the scheme + host **only** (the `/api/v1/...` part is already in each request).

   | Environment | `BASEURL` value |
   |---|---|
   | Local LAN | `http://192.168.0.5` |
   | Localhost | `http://localhost:8080` (use your port) |
   | Production | `https://api.websankul.com` |

2. **Collection-level Bearer auth with `{{token}}`** â€” every authenticated request is set to **Inherit auth from parent**, so it automatically sends `Authorization: Bearer {{token}}`. There is no static token field to edit per-request. Public endpoints (login, OTP, token refresh, webhooks) are explicitly **No Auth**.

3. **Auto-login token capture** â€” the login request has a **Tests** script that reads the response and writes the JWT into the collection variable `{{token}}` (and `{{refreshToken}}` when present). Run it once and every other request is authenticated.
   - Admin: **Auth â†’ POST Admin Login**
   - Client: **Auth â†’ POST Validate OTP (Login)**

## Setup (3 steps)

1. **Import** both `*.postman_collection.json` files into Postman.
2. Create an **Environment**, add a variable `BASEURL`, set its value (see table above), and select that environment.
3. Run the **login** request for the collection. The token is saved automatically; fire any other request next.

## Notes

- Path params appear literally as `:id`, `:planId`, etc. â€” Postman exposes these in the request's **Path Variables** tab; fill in real IDs there.
- `multipart/form-data` requests (uploads) have file fields preset as **File** type â€” pick a file in the form-data editor before sending.
- Request bodies are realistic samples; replace placeholder IDs/values as needed.
- The token-capture script tolerates several response shapes (`data.token`, `accessToken`, `tokens.access`, etc.). If your login response uses a different field, adjust the script in the login request's **Tests** tab â€” it logs a warning to the Postman console when no token field is found.

## Regenerating

The collections are generated by a script so they can be rebuilt as routes change:

```bash
node scripts/build-postman.cjs
```

Endpoint data lives in `scripts/postman-data-admin.json` and `scripts/postman-data-client.json`; edit those and re-run.
